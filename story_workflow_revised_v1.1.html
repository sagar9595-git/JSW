📋 Workflow Proposal
Story & Technical Story
Workflow Redesign
A leaner, more intuitive Jira status flow enhancing the existing workflows — with embedded quality gates and smart transition screens that guide teams through DoR, DoI, and DoD checkpoints at the right moment. Updated after internal alignment session.
👥 Jira Admin Team + RTE 📅 May 2025 · v1.1
ℹ️
This is a first proposal from the Jira Admin team and RTE for stakeholder illustration. It reflects the post-internal-review version (v1.1) incorporating agreed changes: addition of In Analysis status, revised DoI and DoR criteria, and refined transition screen designs.

🎯 Goals of this initiative

The existing SFDC workflows for Story and Technical Story have grown complex. This initiative targets three outcomes simultaneously:
⚡
Leaner flow

Every status must have a distinct, unambiguous meaning with a named owner. No redundant or overlapping states. The new In Analysis status makes BA/refinement work visible.

📈
KPI visibility

Each status is positioned so lead time, cycle time, block time, and test turnaround are measurable with zero manual effort from Jira control charts and CFD.

🔷
Embedded gate screens

Quality checkpoints appear as contextual Jira transition screens — teams are guided through DoR, DoI, and DoD checklists inline, not via separate documents.

🖥 What are transition screens?

A Jira transition screen is a dialog that appears when a team member moves a story between specific statuses. The user must complete the form before the transition is confirmed. This proposal introduces four transition screens:
🟡
DoR Screen
Triggered when moving to Ready for Implementation. Team must checkmark Definition of Ready criteria — including QA readiness — before the story enters a sprint.
🔷
DoI Screen
Triggered when moving to Ready for Testing. Developer confirms functional testing, unit test coverage, staging deployment, merged PR, and license clearance for new dependencies.
🧪
Test Evidence Screen
Triggered after In Test → Ready for Acceptance. QA confirms all test steps are documented and evidence is attached before the Product Owner is invited to UAT.
✅
DoD Screen
Triggered before Closed. All Definition of Done items shown as a checklist — every box must be ticked before the story can be permanently closed.

📦 Issue types in scope

📖
Story
User-facing business functionality

Represents a unit of user value aligned to a D365 CRM sales, service, or marketing scenario. Written from the perspective of a business user. Follows the full workflow including all gates.

⚙️
Technical Story
Non-user-facing technical enabler

Represents infrastructure, integration, refactoring, or platform work. Same workflow — DoR, DoI, and DoD apply with equal rigour. No "tech bypass" on quality gates.

💡
Applies to both Story and Technical Story issue types. The In Analysis status (new) sits between New and DoR, making BA and refinement work explicitly visible on the board and measurable in lead time. Click any flow node to open its transition screen mockup.

📚 Status definitions

🔍 In Analysis Jira: In Analysis Owner: BA / Product Owner KPI: Lead time

Story has been accepted for refinement. The BA and/or Product Owner are actively analysing the requirement — writing acceptance criteria, identifying dependencies, assessing feasibility, and defining testability. A story stays in In Analysis until it is ready to be assessed against DoR criteria.

Activities in this status
✅  Acceptance criteria being drafted
✅  Dependencies identified and owners contacted
✅  Testability assessed — testable vs not testable
✅  Test scope estimated (number of test cases)
✅  Technical feasibility discussed with developer
✅  Story splitting considered if too large
Transitions
→  Ready for Implementation (after DoR gate)
←  New (if de-scoped or split required)

Bottleneck signal:
Long In Analysis = requirement ambiguity or BA capacity constraint
📥 New Jira: New Owner: Product Owner KPI: Lead time

Story has been created and captured in backlog, but active refinement has not started yet. This status is used for intake, initial triage, and prioritization readiness.

Activities in this status
✅  Backlog item created with clear summary
✅  Issue type and team assignment validated
✅  Priority and business value reviewed
✅  Initial dependency signals flagged
Transitions
→  In Analysis (active refinement starts)
←  Cancelled/Archived (if dropped by PO)

Bottleneck signal:
Long New queue indicates intake overload or weak prioritization cadence
🚀 Ready for Implementation Jira: Ready for Implementation Owner: Developer KPI: Cycle starts

Story has passed DoR and is implementation-ready. Scope, acceptance criteria, and testability are clear enough for execution planning and sprint commitment.

Activities in this status
✅  DoR checklist completed and approved
✅  Story pulled into sprint backlog
✅  Implementation approach confirmed
✅  Dev ownership agreed
Transitions
→  In Progress (coding/config starts)
←  In Analysis (if scope re-opened)

Bottleneck signal:
Queue growth here implies sprint capacity mismatch
⚙️ In Progress Jira: In Progress Owner: Developer + Reviewers KPI: Cycle time

Active development, peer review, and implementation refinement happen in this status. Any blocker discovered here should route to On Hold with a blocker log.

Activities in this status
✅  Code/configuration work in progress
✅  PR raised and reviewed
✅  Unit/component tests updated
✅  Technical comments addressed
Transitions
→  Ready for Testing (after DoI screen)
→  On Hold (if blocked)

Bottleneck signal:
Long dwell time suggests rework, review churn, or hidden dependencies
🧪 Ready for Testing Jira: Ready for Testing Owner: QA Engineer KPI: Test TAT starts

Implementation is complete and handed over to QA after DoI validation. The story waits in QA queue until execution begins.

Activities in this status
✅  DoI checklist validated
✅  Test environment availability confirmed
✅  Test cases reviewed/linked
✅  QA handoff acknowledged
Transitions
→  In Test (execution starts)
→  On Hold (env/dependency blocker)

Bottleneck signal:
Growing queue indicates QA capacity or environment bottlenecks
🔬 In Test Jira: In Test Owner: QA Engineer KPI: Test TAT

QA is actively validating acceptance criteria, regression impact, and defect outcomes. This status is execution-intensive and should produce clear evidence trails.

Activities in this status
✅  AC-based tests executed
✅  Regression checks completed
✅  Defects logged and triaged
✅  Evidence captured for handoff
Transitions
→  Ready for Acceptance (with test evidence)
←  In Progress (if defect found)
→  On Hold (if blocked)

Bottleneck signal:
Repeated bounce-back indicates quality gate weakness upstream
👁 Ready for Acceptance Jira: Ready for Acceptance Owner: Product Owner KPI: UAT TAT

QA execution is complete and evidence has been submitted. The story is now waiting for business acceptance, UAT sign-off, and closure readiness checks.

Activities in this status
✅  PO reviews delivered behavior
✅  Business validation notes recorded
✅  UAT sign-off captured
✅  DoD closure prep initiated
Transitions
→  Closed (after DoD screen)
←  In Progress (if UAT rejects)

Bottleneck signal:
Long acceptance delays indicate stakeholder response lag
✅ Closed Jira: Closed Owner: Release Manager / PO KPI: Throughput

The story is fully done, accepted, and closed after DoD completion. All required evidence and release artifacts are in place for traceability.

Activities in this status
✅  DoD checklist completed
✅  Release notes updated
✅  Deployment/status confirmation recorded
✅  Ticket frozen for audit trail
Transitions
✓  Terminal workflow state
↺  Reopen only via governance exception

Bottleneck signal:
High reopen rate indicates poor DoD rigor
⚠️ On Hold Jira: On Hold Owner: Scrum Master KPI: Block time

This is a controlled pause status for active blockers. Entry requires blocker logging, ownership, and an expected resolution path before work can resume.

Activities in this status
✅  Blocker reason captured
✅  Dependency owner and ETA documented
✅  Escalation path defined
✅  Daily unblock follow-up tracked
Transitions
→  Return to previous active status when unblocked
!  Escalate if blocker exceeds SLA window

Bottleneck signal:
Rising hold-time trend highlights systemic dependency management issues

Jira Transition Screen · DoR
Definition of Ready Checklist
🔍 In Analysis → 🚀 Ready for Implementation PO / Scrum Master only
⚠️
All criteria must be checked before this story can enter a sprint. Unchecked items block the transition. The Jira transition will fail if any box is unticked.
🔷 Definition of Ready — All items required to proceed
Readiness checklist
0 / 7
📋 Acceptance & Estimation
✓
Acceptance criteria written in Given / When / Then format and attached to the story description
✓
Story points estimated via team planning poker — consensus reached and recorded
🔗 Dependencies & Design
✓
Dependencies identified — owner named and delivery date confirmed in Jira dependency field
✓
Technical approach agreed — design notes or architecture decision record (ADR) attached to the ticket
🧪 QA Readiness
✓
Testability assessment completed — story is classified as Testable or Not Testable, with justification noted in the ticket
✓
Test case count estimated — expected number of test cases documented (e.g. "3 positive, 2 negative, 1 edge case") to allow QA capacity planning
If Not Testable: clearly state why (e.g. infrastructure-only change with no functional behaviour) and confirm with QA Lead before proceeding.
✅ Sprint Commitment
✓
Product Owner formally commits this story to the sprint — sprint field confirmed in Jira
Additional notes (optional)
Jira Transition Screen · DoI
Definition of Implemented Checklist
⚙️ In Progress → 🧪 Ready for Testing Developer / Tech Lead only
🔷
The developer confirms that the implementation is complete, self-tested, and ready for independent QA. Do not hand over a story with an untested integration path, a failed build, or uncleared external dependencies.
🔷 Definition of Implemented — Developer confirmation required
Implementation checklist
0 / 6
🧑‍💻 Developer Self-Testing
✓
Functional testing by developer completed on the individual developer sandbox — all acceptance criteria scenarios manually verified before handoff to QA
✓
Unit test code coverage ≥ 80% — coverage report generated and result noted below (attach or paste coverage % per class/method)
Unit test coverage report must be attached or linked. Coverage below 80% blocks this transition unless a Tech Lead exception is recorded.
🚀 Deployment & Code Review
✓
Deployment and staging procedures recognised — deployment runbook reviewed, staging environment confirmed, and deployment steps documented or referenced
✓
PR is merged after review — pull request approved by minimum two peer reviewers and merged to the release branch; build pipeline fully green
📦 External Technical Dependencies (complete only if applicable)
✓
If a new external technical dependency was used (e.g. unlocked package, third-party library, external API):
→ License clearing has been passed — legal/IP approval recorded (reference or ticket number below)
✓
New dependency added to the team's table of dependencies — name, version, license type, purpose, and approved-by recorded in the dependency register
If no new external dependency was introduced in this story, tick both items above and note "N/A — no new external dependency" in the notes field below.
Unit test coverage % (required)
e.g. Overall: 84% · Class A: 91% · Class B: 78%...
License clearing reference / dependency register link (if applicable)
